JMeter 03--并发测试

Posted on 2019-10-16 | In JMeter

环境信息

目标

设置并发数，收集性能数据；

线程设置

位置：线程组单击；

执行时长：右上角有脚本的执行时间，用于统计脚本的执行时长；

线程属性中最重要的几个参数：
线程数：也就是我们提到的并发数（类似LoadRunner中的VUser）;

Ramp-Up时间：在多长时间内启动所有的线程数。举例，若线程数为10，Ramp-Up时间为5，则表示JMeter会在5秒内启动10个线程（每秒启动2个），来执行这个线程组下所有请求；

循环次数：永远表示会一直执行下去，直到手动停止执行。若取消勾选永远，在后面的输入框中输入对应的值，则表示每个线程执行完对应的次数后，自动退出；
注意：这里不是所有的线程执行次数的总和，而是单个线程需要执行的次数；

聚合报告

需求统计接口执行时的响应数据，我们需要添加聚合报告模块；
位置：线程组右键–添加–监听器–聚合报告；

设置线程数为10，Ramp-Up时间5秒，循环次数为2，启动执行，结果如下：

重点指标：
样本：循环次数为定量时，关注样本数与实际设置的循环次数是否一致；

95%百分位|99%百分位：这两个指标反应的是接口的响应时间（单位：毫秒），是我们性能测试中最重要结果；
所有与时间相关的单位均为毫秒；

异常%：短时间压力测试，异常%应该为0，长时间压力测试后，异常%应该在我们可接受的范围内或为0；

网络吞吐：性能测试不达标或不理想的情况下，我们可以监控压力机和服务器的网络带宽使用情况，避免因此而产生的性能瓶颈；

JMeter 02--参数化

Posted on 2019-10-16 | In JMeter

环境信息

目标

能实现接口间动态参数关联。

设置全局参数

到此为止，我们登录接口已经能执行成功，并且能查看接口的请求和返回，但还有一个需要解决：获取登录后Token；
登录成功后，我们需要将Token传递给下一个请求使用；

正则表达式提取器

我们先添加一个正则表达式提取器，位置：Http请求右键–添加–后置处理器–正则表达式提取器；

正则表达式提取设置页：
需要根据请求的实际返回进行修改正则表达式；
关于正则表达式提取的，这里不展开，JMeter官方正则文档点这里
关键语句（注意标点中英文状态）：“token”:”(.*?)”

BeanShell后置处理器

我们再添加一个BeanShell后置处理器，位置：Http请求右键–添加–后置处理器–BeanShell后置处理器；

添加这个后置处理器是为了将获取到的变量存储到全局变量中；
其中getToken是前面通过正则表达式获取到的变量值，pubToken是全局变量的key，后续可以通过这个key获取到他的值；
关键语句：${__setProperty(pubToken,${getToken},)}

使用全局参数

上面我们获取到Token并存储到全局变量后，我们后续的请求就需要用到了，我们需要将获取到的Token带入后续请求的头信息中，否则系统鉴权不通过，无法正常访问对应接口；
pubToken是我们上面存储全局变量值，这样我们就实现了跨线程间动态传递参数；
关键语句：${__property(pubToken)}

MERGE

Postman API自动化09--执行利器Runner

Posted on 2019-06-26 | In API自动化

环境信息

目标

更方便地调试接口测试脚本；

快速了解

批量执行入口

Runner界面预览

功能详解

Enviroment：我们可以通过选择不同环境变量，让脚本在不同环境上执行，满足在各类环境上执行测试脚本的需求；

Iterations：执行次数，我们需要脚本运行多少次，非并发执行，是一次一次执行；

Delay：每一次请求执行之前暂停的时间，对于异步处理的请求可能需要添加这个等待时间；

Log Responses：记录哪些请求的日志；

Data：从外部文件引入数据（数据参数化的一种实现）；

Keep variable values：保持（或者叫更新）本次执行时产生的变量至环境变量和全局变量。我们可以在执行之前清理环境变量和全局变量的目前值，执行完成之后观察值的变化；

Save cookies after collection run：默认选择（还没搞清楚具体作用）；

RUN ORDER
RUNNER 页面右侧的 RUN ORDER 对于部分调度非常有帮助；
我们可以把他理解为PlayList，在这里我们可以调整接口执行顺序、选择要执行的接口；

执行结果

执行完成后，我们可以点击接口用例的名称，显示此次接口请求的相关内容：请求地址、请求头、请求体、返回头、返回体；
查看请求内容和前面讲到的Console输出，是我们调试接口测试的常用手法；

我有多个接口要调试，有些接口之间有依赖关系，这种情况我应该怎么办？
没问题！Postman在多方面均解决了这个问题；

通过【Runner】执行时，我们勾选【Keep variable values】，此时也会更新并记录你的变量值（环境变量和全局变量）；
