方法名 返回值类型 功能说明
maximize() void 最大化浏览器
fullscreen() void 设置浏览器全屏状态，不常用

refresh() void 刷新，相当于浏览器的刷新按钮功能
quit() void 退出当前浏览器

页面

方法名 返回值类型 功能说明

getPageSource() String 获取并返回当前页面源代码
getWindowHandle() String 获取并返回当前页面源代码
getWindowHandles() Set 获取并返回当前浏览器所有窗口句柄，getWindowHandles()后续将用于手动切换页面
close() void 关闭当前页面，如果当前只有一个页面则退出浏览器

元素

方法名 返回值类型 功能说明
findElement() WebElement 寻找并返回找到的第1个元素，若匹配到多个元素也返回第1个
findElements() List 寻找并以List<WebElement>返回元素列表，获取元素时从下标0开始
sendKeys() String 输入文本（单/多行文本框）
click() String 点击（链接/按钮/图片）

业务安全测试切入点

Posted on 2016-04-18 | In 测试

题记

[放在互联网的数据就不是隐私]，每一个互联网人对此都深有感触
互联网企业内部管理不善导致数据出卖、安全漏洞被黑客盗走数据的事情时有发生
2016年频繁出现的互联网安全事故，直接导致国内知名漏洞报告平台乌云被查封
出现一系列的事故之后，公司了也展开了安全测试，包括服务安全和业务安全
这里大致列举日常测试工作中应该要注意的业务安全问题

XSS类：包括存储型和反射型

反射型最容易传播的是搜索URL，注意处理处理这类URL中的XSS风险代码，其中存储型的危害比反射型的危害相对要更大，建议在平台做统一处理
测试代码举例：http://drops.wooyun.org/tips/845 [乌云已挂出维护公告]

前端存在业务校验

任何前端的检验都是为提高用户体验而做，不具体任何安全性。若涉及业务上重要的校验，需要确保前后端都存在统一的逻辑校验
例如：数字大小范围、时间长度范围

POST表单冗余数据

POST表单中提交了页面或业务逻辑中不允许直接修改的数据
例如：常用的如手机号、邮箱，修改时会单独跳转到其他页面进行校验，展示时以灰色不可编辑的形式展示在表单中

突破搜索参数数量

很多网站都会有搜索，搜索中有很多的条件可选，业务在设计时对部分搜索条件的数量进行了限制
在截获请求之后，对搜索条件进行扩充后并发请求服务器，会对服务器造成比较大的压力，这一条也与前面讲到的<前端存在业务校验>有关联

密码类传输

主要涉及注册、登录、修改密码、忘记密码这几类常用操作，传输过程中若使用明文密码是非常不安全的

数据操作类带有ID

密切注意编辑、删除操作中带有数据ID的情况，水平越权操作数据，是非常危险的漏洞

数据库存储长度

在产品设计时会对数据的可输入在前端做一个长度或输入类型限制，是为了保证更好的用户体验和确保有限的系统开销
在前面我们已经知道前端的限制是没有任何安全意义的，通过工具修改数据类型和长度同样可被正常提交，若出错很有可能会暴露数据库的关键信息

重复提交请求|数据数量有限制

对于不允许重复提交的请求和数量有限制的存储，不光在前端做限制，在服务端也在有对应的校验和限制

数据回填

将用户输入的数据直接回填在JSP，然后作为标签中某个属性值显示在页面的HTML中，这种情况下如果不作处理，可能有XSS
常见有属性有value和title

自动登录

修改密码、重置密码类操作时，一定不要在最后附加一个自动登录的动作。不可将这个判断将给前端或者浏览器

撞库

网络上有很多泄漏的账号密码资源，被拿到之后会用于撞库，取得一些网站的用户名密码
这各情况通常的做法是增加操作成本，比如限制每个IP下请求对应接口的次数、出现验证码、锁定账户2小时等

暴力破解

重灾区在找回密码验证码，或者其他的任何重要操作需要数字验证码的部分
建议：将验证码复杂度增加（由纯属4位数据变成6位数字和字母组合），同时加上图形验证码，同时最好有其他访问防爆或监控机制

Selenium Web自动化02--TestNG常用注解

目标

了解TestNG常用注解及其执行顺序

基本注解

最常用

最常用的几个注解：
@Test、@BeforeTest、@BeforeClass、@AfterTest、@AfterClass

指定依赖关系：在@Test注释中使用属性dependsOnMethods
1.以特定顺序调用测试用例中的方法，或者可能希望在方法之间共享一些数据和状态
2.如果依赖方法失败，则将跳过所有后续测试方法

结论

注解的执行顺序如下：
@BeforeTest
@BeforeClass
@Test
@AfterClass
@AfterTest

这里面试的时候可能会问到@BeforeTest和@BeforeClass谁先执行，不是简单的先后，而要区分Before还是After，执行先后关系有一定的对称关系。
也可能会问到BeforeMethod和@AfterMethod是怎么生效的，这对注释会在每一个@Test的开始和结束执行。但会在@BeforeTest和@BeforeClass之后，@AfterClass和@AfterTest之前，执行先后也有对称关系。

使用dependsOnMethods声明依赖关系的方法，并不会在依赖方法执行后马上执行，其中可能先执行另外的方法，具有一定的随机性。
如果有明确的执行依赖顺序，需要对所有方法均设置好依赖关系。

时态的使用和链接位置的控制

请看下图（QQ试用版过期的提示）

存在的问题：
“将”的意思是未来某个时候，不是现在
“将无法继续使用”，意思是在以后某个时间开始无法使用，而这里的实际意思是“已无法继续使用”。看到将无法继续使用，我认为还是可以用的，尝试多次其实已经无法使用这个试用版本登录。
（博主每天都使用QQ，能断定这是第一天弹出来的提示信息）

“请您到官网下载最新版本”，这里有一个链接，链接的位置挂在“官网”两个字，实际把链接由“官网”换成“下载最新版本”更合适。
为什么说换成“下载最新版本”更合适？
原因有两个：一是“官网”两个字的可点击区域过小而且换行；二是我的版本过期了，不应该是去官网，而应该是下载最新可用版本，所以链接为“下载最新版本” 更佳。

软件测试基础流程

Posted on 2013-03-14 | In 测试

软件测试工作中最重要的三要素：需求+技术+流程
下图是Tester Chen整理的软件测试基础流程图：

需求评审会

由产品人员或需求人员主持，主要讨论需求的完整性、合理性，同时加深项目组各个成员对需求的理解

测试用例评审会

在测试开始前由测试工程师组织，主要确认测试用例对于需求的覆盖度、测试用例本身的合理性

测试评审会

测试工作完成后，在上线之前由测试工程师组织的一次评审会议，主要内容是再次讨论测试是否完全、根据产品或项目现测试情况是否符合上线条件

执行测试&报告回归缺陷

这个阶段比较特殊，在项目中我们一般分为α测试和β测试，在产品中我们一般分为开发环境测试、预上线环境测试
α测试大多数情况下执行者为测试工程师，β测试大多数情况下为目标用户
开发环境测试，是指测试工程师在开发环境下进行的测试，可随时更新、修改代码
预上线环境测试：是指开发环境测试通过后，使用线上真实数据进行测试的过程，主要为验证测试结果，保证生产上线的正确性
