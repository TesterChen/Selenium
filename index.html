Git03--分支合并

Posted on 2019-07-03 | In Git

目标

能将分支代码合并到master；
处理合并冲突；

客户端合并

通过Git客户端命令操作合并，常用的有merge和rebase；

MERGE
1、提交分支所有改动至远程仓库对应分支(branch_name)；
2、切换至master分支，并更新master代码(git pull)；
如果本地的master分支代码比仓库的代码要旧，那么提交的时候会抛异常，我们仍然需要pull，建议先pull，再进行第3步的合并；
3、合并分支代码至master(git merge <branch_name>)；

在线PR合并

通过Github操作，生成Pull request（简称PR），在线进行合并；

团队成员在Push分支代码后，Github仓库主页能看到对应提交；

点击上图中的【Compare & pull request】按钮，即可进入下图状态；
我们可以清晰的看到，是从chenjp分支合并代码到master分支；
同时看到“Able to merge，These branches can be automatically merged”，表示我们chenjp分支的代码可以自动合并至master；

在页面的下方有修改的对比显示，方便我们确认修改；

点击上图中的【Create pull request】就生成了PR，在Pull request选项卡中我们可以看到；

进入PR详情，我们可以看到这次PR的详情，以及操作合并的按钮【Merge pull request】；

Merge完成后，我们再返回仓库首页，看到最后一次提交日志和提交时间已经更新了，与我们Pull的时间、日志一致；
说明代码合并操作成功；

Postman API自动化09--执行利器Runner

Posted on 2019-06-26 | In API自动化

环境信息

目标

更方便地调试接口测试脚本；

快速了解

批量执行入口

Runner界面预览

功能详解

Enviroment：我们可以通过选择不同环境变量，让脚本在不同环境上执行，满足在各类环境上执行测试脚本的需求；

Iterations：执行次数，我们需要脚本运行多少次，非并发执行，是一次一次执行；

Delay：每一次请求执行之前暂停的时间，对于异步处理的请求可能需要添加这个等待时间；

Log Responses：记录哪些请求的日志；

Data：从外部文件引入数据（数据参数化的一种实现）；

Keep variable values：保持（或者叫更新）本次执行时产生的变量至环境变量和全局变量。我们可以在执行之前清理环境变量和全局变量的目前值，执行完成之后观察值的变化；

Save cookies after collection run：默认选择（还没搞清楚具体作用）；

RUN ORDER
RUNNER 页面右侧的 RUN ORDER 对于部分调度非常有帮助；
我们可以把他理解为PlayList，在这里我们可以调整接口执行顺序、选择要执行的接口；

执行结果

执行完成后，我们可以点击接口用例的名称，显示此次接口请求的相关内容：请求地址、请求头、请求体、返回头、返回体；
查看请求内容和前面讲到的Console输出，是我们调试接口测试的常用手法；

Postman API自动化06--团队协作

Posted on 2019-06-13 | In API自动化

环境信息

目标

接口测试脚本往往不是由一个同事在开发和使用，而是一个团队；
通过Postman自带功能，实现团队协作，包括脚本合并、编辑和执行。

创建工作区

点开顶部工作区的下拉，即可创建工作区；

根据页面填写必要信息即可创建工作区，同时还可邀请需要加入的同事（注意：这个同事也需要注册Postman账号）

成员邀请

在创建工作区时可以邀请同事，也可以在切换到对应工作区之后，通过【Invite】功能邀请同事加入；

设置权限

团队中不是每一位成员都需要编辑测试脚本，为了避免误操作，我们需要设置权限；
Team Workspace下每一个Collection可以针对Team成员单独设置权限，非常灵活；
点击Collection右下角的【…】功能菜单，点击【Manage Roles】

通过设置，我们可以控制成员的权限（可读还是可写），从而避免误编辑引起的协作问题；

注意：如果我们把【Everhyone in Team】设置为Editor，那么下方再单独设置为Viewer是不生效的，页面会有提示；

分支和合并

Postman引入了分支和合并的概念：Frok和Merge；
在Team Wrokspace对应Collection的功能菜单中选择【Create a fork】即可创建一个属于自己的分支；

创建分支时可以选择分支存储的Workspace（建议存储到自己的Space）；

在分支上进行修改后，我们可以进行合并操作，在Wrokspace对应Collection的功能菜单中选择【Merge changes】;

合并时有可能需要Pull操作，因为在我们合并之前可能已经有其他人Merge或者原Team Workspace内容被有Editor权限的用户直接修改了；

Pull完之后，可以点击同步按钮，观察自己Workspace中Collection的变化，如果正常，则再次操作Merge，直接合并到Team Workspace即可；

直接修改

上面提到的分支、合并，是开发流程中的节点，有一定的复杂度，但数据安全；
我们也可以有另一种选择，直接编辑Team中测试脚本，不开分支、合并，这种数据不安全；
使用直接修改模式时，建议定期导出Json文件或者Fork分支到个人工作区保存，避免数据误操作；

模式

Postman有两种模式：在Postman右下方可以看到【Browse】和【Build】；

当我们在切换到Team Workspace时默认为【Browse】模式，Browse模式下点击Collection时会跳转到Web浏览器，不是很方便；
Team Workspace下我们可以手动切换到【Build】模式，在这个模式下：
如果【Manage Roles】中你是Editor权限，那么你可以直接编辑、保存、同步脚本至Team Workspace，而不用通过Frok和Merge的形式提交变更（但然不建议多人修改同一个脚本）；
如果你只是Viewer，虽然没有保存脚本权限，但可以更方便地执行脚本，同时实时更新最新脚本也会变更更容易；

科学上网

Postman是谷歌系的产品，部分服务可能无法直接访问（如：Merge功能）；
当你发现部分功能使用或链接无法访问时，你可能需要一个科学上网的工具；

被邀请加入团队之后，Postman会给你的邮箱发送一封确认加入邮件，如果收件箱没有找到，记得去垃圾邮件中找找；

使用体验

在团队内部使用了Team协作功能，可以说确实很好用；
不过，免费的空间很小（25个请求），对于联调数量不多的接口问题不大，涉及到大批量的测试脚本管理就显得很不够了；
突破限制我们需要按团队人数/年付费；

临时解决办法：
Postman可以说是接口测试工具中非常好用的，为了继续使用这个工具，又满足团队协作，我们决定使用GitHub来配合；
Postman脚本导出之后为json文本信息，可以被版本管理工具识别变更，这就很好了；
编辑完脚本信息之后导出为json文件，提交至GitHub，其他人随时可以从GitHub拿到最新的脚本文件，导致Postman即可；
可以提升大家的Git技能，虽然麻烦些，也不算亏；

环境变量：顾名思义与环境有关，如：不同环境执行测试脚本时配置的用户名、密码应该被放在环境变量中存储；

作用范围

1、环境变量与全局变量的作用范围为Wrokspace，不同Wrokspace之间的变量相互隔离，互不影响；
2、当环境变Key与全局变量Key重复时，优先使用环境变量中的值，即使环境变量中对应Key的值为空；
